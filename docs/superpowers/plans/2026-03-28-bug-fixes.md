## Task 2: Fix Newsletter Subscription Check

**Files:**
- Modify: `apps/accounts/models.py:46-47`

**The bug:** `hasattr(self, 'newsletter_subscription')` never triggers a DB query â€” it only checks if the attribute is already loaded in memory. When the user object hasn't been fetched with `select_related('newsletter_subscription')`, this always returns `False` even if the user is subscribed. The Subscribe button shows incorrectly.

- [ ] **Step 1: Replace the method**

Open `apps/accounts/models.py`. Find lines 46-47:

```python
def is_newsletter_subscribed(self):
    return hasattr(self, 'newsletter_subscription')
```

Replace with:

```python
def is_newsletter_subscribed(self):
    try:
        return self.newsletter_subscription is not None
    except NewsletterSubscriber.DoesNotExist:
        return False
```

`newsletter_subscription` is a `OneToOneField` reverse relation. Django raises `RelatedObjectDoesNotExist` (a subclass of `NewsletterSubscriber.DoesNotExist`) when the related row doesn't exist â€” that's the correct signal. This also works when prefetched, so no regression there.

- [ ] **Step 3: Commit**

```bash

git commit -m "fix: correct newsletter subscription check using try/except

hasattr() only checked Python object state, not the database.
```

## Task 3: Fix MyArticlesView â€” Use DB Queries Instead of Python Filtering

**Files:**
- Modify: `apps/articles/views.py:37-48`
- No template changes needed (`my_articles.html` already uses `{{ drafts.count }}` which works on QuerySets)

**The bug:** The view fetches ALL of the user's articles into memory, then splits them with Python list comprehensions. For a prolific author this is slow and uses excessive memory. Also, `{{ drafts.count }}` in the template calls `list.count()` (which requires an argument), so tab badges silently show empty â€” a secondary bug caused by the same issue.

- [ ] **Step 1: Rewrite `MyArticlesView.get_context_data`**

Open `apps/articles/views.py`. Find lines 37-48:

```python
class MyArticlesView(AuthorRequiredMixin, TemplateView):
    template_name = 'articles/my_articles.html'

def get_context_data(self, **kwargs):
        ctx = super().get_context_data(**kwargs)
        user = self.request.user
        all_articles = Article.objects.filter(author=user).select_related('category').order_by('-updated_at')
        ctx['drafts']    = [a for a in all_articles if a.status == 'draft']
        ctx['in_review'] = [a for a in all_articles if a.status == 'in_review']
        ctx['revisions'] = [a for a in all_articles if a.status == 'revision']
        ctx['published'] = [a for a in all_articles if a.status == 'published']
        return ctx
```

Replace with:

```python
class MyArticlesView(AuthorRequiredMixin, TemplateView):
    template_name = 'articles/my_articles.html'

    def get_context_data(self, **kwargs):
        ctx = super().get_context_data(**kwargs)
        user = self.request.user
        base = Article.objects.filter(author=user).select_related('category').order_by('-updated_at')
        ctx['drafts']    = base.filter(status='draft')
        ctx['in_review'] = base.filter(status='in_review')
        ctx['revisions'] = base.filter(status='revision')
        ctx['published'] = base.filter(status='published')
        return ctx
```

Each `.filter()` call on `base` produces a new QuerySet â€” Django doesn't execute any query until the template iterates them. `{{ drafts.count }}` in the template now calls `QuerySet.count()` (zero-arg, correct), fixing the empty badge bug as a side-effect.

- [ ] **Step 2: Manual verify**

1. Log in as an Author who has articles in multiple statuses.
2. Go to `/articles/my/`.
3. Check each tab â€” the badge counts (Drafts, In Review, etc.) should show correct numbers.
4. Articles should appear correctly under each tab.

- [ ] **Step 3: Commit**

```bash
git add apps/articles/views.py
git commit -m "fix: filter MyArticlesView articles in DB, not Python memory

List comprehensions loaded all articles into memory and caused
QuerySet.count() to silently fail (list.count() requires an argument).
Each status now uses a separate DB-filtered QuerySet."
```

## Task 4: Fix Slug Generation Race Condition

**Files:**
- Modify: `apps/articles/views.py:127-134` (inside `ArticleCreateView`)

**The bug:** The `_unique_slug` method checks if a slug exists then assigns it â€” but between the check and the save, another concurrent request could assign the same slug. This causes an `IntegrityError` (unique constraint violation) on the save. The fix: catch `IntegrityError` and retry with a new slug.

- [ ] **Step 1: Add `IntegrityError` import**

At the top of `apps/articles/views.py`, `django.db` is not currently imported. Add `IntegrityError` to the imports. Find the existing imports block at the top and add:

```python
from django.db import IntegrityError
```

Full updated import block (add this line after the existing django imports):

```python
import os
from django.contrib import messages
from django.contrib.auth.mixins import LoginRequiredMixin
from django.core.exceptions import PermissionDenied
from django.core.files.storage import default_storage
from django.db import IntegrityError
from django.db.models import Count
from django.http import JsonResponse
from django.shortcuts import get_object_or_404, redirect, render
from django.utils.text import slugify
from django.views import View
from django.views.generic import ListView, DetailView, CreateView, UpdateView, TemplateView
from django.urls import reverse, reverse_lazy
```

- [ ] **Step 2: Rewrite `form_valid` and `_unique_slug` in `ArticleCreateView`**

Find `ArticleCreateView` (lines 112-134). Replace `form_valid` and `_unique_slug`:

```python
class ArticleCreateView(AuthorRequiredMixin, CreateView):
    model = Article
    form_class = ArticleForm
    template_name = 'articles/article_form.html'

    def form_valid(self, form):
        article = form.save(commit=False)
        article.author = self.request.user
        article.status = 'draft'
        base_slug = slugify(form.cleaned_data['title'])
        article.slug = base_slug
        counter = 1
        while True:
            try:
                article.save()
                break
            except IntegrityError:
                article.slug = f'{base_slug}-{counter}'
                counter += 1
        form.save_m2m()
        messages.success(self.request, 'Article saved as draft.')
        return redirect('articles:detail', slug=article.slug)
```

Also delete the `_unique_slug` method entirely â€” it is no longer used.

The while loop attempts the save. If another request grabbed the same slug between our check and our save, the DB raises `IntegrityError`, we increment the counter and retry. This is atomic â€” the DB constraint is the source of truth, not our pre-check.
